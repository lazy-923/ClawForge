# ClawForge 产品需求文档（PRD）

## 一、项目名称与定位

### 1. 项目名称

**ClawForge**

### 2. 名称含义

`Claw` 延续 OpenClaw / Mini-Claw 的显式技能与本地 Agent 脉络；`Forge` 强调“锻造、提炼、演化”，体现本项目不再停留于静态 Skill 文件，而是把交互经验逐步锻造成可复用、可治理、可演化的技能资产。

### 3. 项目一句话定义

ClawForge 是一个面向本地 AI Agent 的 **Skill Gateway + Skill Evolution** 工作台，用文件驱动的方式把“技能发现、技能激活、技能草稿生成、技能合并提升”串成一条透明、可编辑、可审查的闭环。

---

## 二、项目背景

### 1. 当前问题

Mini-Claw 已经具备：

- 文件驱动 Prompt
- `SKILL.md` 技能机制
- `MEMORY.md` 长期记忆
- 工具调用可视化
- 本地会话持久化

但当前 Skill 机制仍然存在明显短板：

1. **技能激活方式过于被动**
- 当前主要依赖主 Agent 阅读 `SKILLS_SNAPSHOT.md` 后自行决定要不要读取某个 `SKILL.md`
- 缺少独立的 Skill Gateway，无法对技能做专门的检索、筛选和注入

2. **技能库是静态的**
- 现在的技能主要靠手工编写
- 用户长期交互中暴露出的稳定偏好、约束和流程，无法沉淀为显式技能资产

3. **经验沉淀停留在 memory 层**
- 当前系统能记住信息，但难以把“以后应该怎么做”升级成结构化、可执行的行为规范

4. **缺少技能治理**
- 缺少技能草稿、版本、来源、合并、淘汰等机制
- Skill 数量一多，Prompt 会膨胀，技能也容易重复与污染

### 2. 引入新方向的目标

本项目希望在保留 Mini-Claw 核心优点的基础上，引入一套更完整的技能中枢能力：

- 让 Skill 从“静态说明书”升级为“动态可演化工件”
- 让技能激活从“主 Agent 自行猜测”升级为“Gateway 检索与注入”
- 让经验沉淀从“原始记忆片段”升级为“技能草稿与技能资产”

---

## 三、产品目标

### 1. 总目标

打造一个 **本地优先、文件驱动、透明可控** 的 Agent 技能工作台，使系统具备以下能力：

1. 根据当前请求自动检索最相关技能，而不是把全部技能硬塞进 Prompt
2. 从用户交互中自动生成候选技能草稿
3. 对技能草稿执行 add / merge / ignore 的治理流程
4. 保持整个技能生命周期可观察、可编辑、可回滚

### 2. 非目标

本阶段明确不做：

- 自动无人值守地全量改写主技能库
- 复杂多租户 SaaS 平台
- 依赖外部向量数据库的重型架构
- 把项目做成单纯的 OpenAI-compatible 代理服务

---

## 四、核心产品理念

### 1. 文件优先，而不是黑盒优先

所有关键状态优先存为本地文件，而不是隐含在系统内部：

- 正式技能
- 技能草稿
- 技能版本历史
- 技能使用统计
- 技能合并记录

### 2. 显式技能，而不是隐式 Prompt 漂移

系统不是偷偷修改系统提示词，而是把能力沉淀为显式工件，允许用户：

- 查看
- 编辑
- 对比
- Promote
- 回滚

### 3. 半自动演化，而不是全自动污染

ClawForge 不追求“每轮对话自动入库”，而是追求：

```text
交互信号 -> 生成 Skill Draft -> 提供治理建议 -> 人工确认 Promote
```

这保证系统既能学习，又不至于失控。

### 4. 本地工作台，而不是纯后端代理

ClawForge 的最终形态是一个本地 Agent 工作台：

- 左侧管理会话与技能
- 中间进行对话与观察技能命中
- 右侧审查和编辑 Skill Draft / Skill / Memory

---

## 五、目标用户

### 1. 第一类用户：Prompt / Agent 开发者

特点：

- 需要不断调试 Agent 行为
- 希望技能体系可审查、可维护
- 不满足于黑盒式“记忆增强”

需求：

- 想知道本轮为什么命中了某个技能
- 想知道某条新技能是从哪轮对话提炼出来的
- 想手工编辑技能，而不是完全信任自动生成

### 2. 第二类用户：重度本地 AI 工作流使用者

特点：

- 长期重复做类似任务
- 有稳定风格、约束和偏好
- 希望 Agent 越用越懂自己

需求：

- 系统能积累“做事方式”，不只是记住“历史内容”
- 不想每次重新描述流程与格式要求

## 六、核心能力范围

## 6.1 Skill Gateway

### 功能目标

在每轮对话前，先通过一条专门的技能激活路径，决定应该给主 Agent 注入哪些技能。

### 核心能力

1. Query Rewrite
- 将用户问题改写为更适合检索技能的查询

2. Skill Retrieval
- 从正式技能库中检索候选技能

3. Skill Selection
- 判断候选技能是否真的适合当前任务

4. Skill Context Injection
- 将少量相关技能注入到 Prompt，而不是全量注入

### 产品价值

- 降低 Prompt 膨胀
- 提升技能命中精度
- 让 Skill 激活过程可解释

---

## 6.2 Skill Drafting

### 功能目标

从真实交互中提取可复用的行为规则，生成候选技能草稿。

### 抽取原则

只提取：

- 稳定要求
- 可复用流程
- 可迁移的风格或约束

不提取：

- 一次性请求
- 临时上下文
- 具体 case 的实体内容

### 输出形态

Skill Draft 应至少包含：

- 名称
- 描述
- Goal
- Constraints / Style
- 可选 Workflow
- 来源会话
- 置信度
- 推荐动作

---

## 6.3 Skill Governance

### 功能目标

对新技能草稿进行治理，而不是直接写入正式技能库。

### 治理动作

1. `add`
- 新增为一项正式技能

2. `merge`
- 合并到已有技能中

3. `ignore`
- 放弃该草稿
### 治理要求

- 给出推荐动作
- 给出相似技能
- 给出为什么要 add / merge / ignore
- 保留人工确认入口

---

## 6.4 Skill Versioning

### 功能目标

让正式技能具备版本历史，而不是每次被覆盖后丢失上下文。

### 核心能力

- 记录 skill lineage
- 记录 merge history
- 支持 patch 级版本递增
- 支持查看前后差异
- 支持回滚

---

## 6.5 Skill Workspace

### 功能目标

把技能系统做成一个可视化工作台，而不是只靠文件夹操作。

### 工作台视图

1. 正式技能列表
2. 技能草稿列表
3. 本轮命中技能
4. 技能版本历史
5. 技能使用统计

---

## 七、关键产品差异化

### 1. 与传统 Memory 系统的区别

Memory 记住的是：

- 事实
- 偏好
- 历史片段

ClawForge 关注的是：

- 行为规范
- 工作流策略
- 可执行技能

### 2. 与纯 Prompt 管理系统的区别

纯 Prompt 管理往往只能编辑静态提示词。

ClawForge 进一步解决：

- 技能检索
- 技能激活
- 技能草稿抽取
- 技能版本演化

### 3. 与 Auto-Skill 的区别

ClawForge 吸收的是“技能演化思想”，但产品形态不同：

1. 以本地工作台为中心，而不是以代理服务为中心
2. 以文件工件为第一存储单元，而不是以运行时服务为第一中心
3. 采用半自动 Promote 机制，而不是默认全自动闭环
4. 更强调前端可视化审查和手工编辑能力

## 八、功能需求

## 8.1 P0 功能

### P0-1 技能网关主链路

- 接收用户消息
- 先做技能查询改写
- 检索相关技能
- 将命中技能注入 Prompt
- 返回命中结果给前端

### P0-2 技能草稿生成

- 每轮对话结束后异步抽取 Skill Draft
- 写入本地 `skill_drafts/`

### P0-3 技能治理建议

- 对每个 Draft 给出：
  - 推荐动作
  - 相似技能
  - 理由说明

### P0-4 前端可观察性

- 显示本轮命中的技能
- 显示新生成的 Skill Draft
- 可以打开并查看 Draft 内容

---

## 8.2 P1 功能

### P1-1 Skill Promote

- 用户可手动将 Draft Promote 为正式技能

### P1-2 Skill Merge

- 用户可将 Draft 合并到相似技能

### P1-3 Skill Versioning

- 记录每次 Promote / Merge 的版本历史

### P1-4 使用统计

- 记录技能被检索次数
- 记录技能被采纳次数

---

## 8.3 P2 功能

### P2-1 技能淘汰建议

- 对长期高召回低使用的技能给出 stale 提示

### P2-2 文档到技能

- 从知识文档中离线提取候选技能

### P2-3 Replay 驱动技能评估

- 对高频技能做离线回放评估

---

## 九、关键用户流程

## 9.1 在线问答流程

```text
用户输入问题
-> Skill Gateway 改写查询
-> 检索技能
-> 选择并注入技能
-> 主 Agent 执行
-> 返回回答 + 命中技能
```

## 9.2 技能沉淀流程

```text
一轮对话结束
-> Draft Extractor 提取候选技能
-> Skill Judge 生成 add / merge / ignore 建议
-> 写入 skill_drafts/
-> 前端展示
-> 用户确认 Promote / Merge / Ignore
```

## 9.3 技能演化流程

```text
用户打开某个 Draft
-> 查看推荐相似技能
-> 执行 Promote 或 Merge
-> 更新正式技能库
-> 写入版本与历史记录
```

---

## 十、成功标准

### 1. P0 成功标准

1. 系统能在聊天前命中并注入少量相关技能
2. 前端能展示本轮命中技能
3. 对话结束后能生成 Skill Draft
4. Draft 可以被查看和审查

### 2. P1 成功标准

1. Draft 可以 Promote 为正式技能
2. Draft 可以 Merge 到已有技能
3. 正式技能保留版本历史

### 3. 产品层成功标准

1. 用户重复描述流程和风格约束的次数明显下降
2. 技能命中质量优于“全量技能快照 + Agent 自行猜测”
3. 技能库能持续增长，但不明显失控或污染

---

## 十一、阶段规划

### 阶段一：Skill Gateway MVP

- Query Rewrite
- Skill Retrieval
- Skill Injection
- 命中技能展示

### 阶段二：Skill Draft MVP

- Draft Extraction
- Draft 存储
- Draft 列表与查看

### 阶段三：Skill Governance MVP

- add / merge / ignore 建议
- Promote / Merge 操作
- 版本与 lineage

### 阶段四：Skill Evolution 增强

- stale skill 审计
- 使用统计
- 离线评估
- 文档到技能

---

## 十二、最终产品描述

ClawForge 不是一个“带记忆的聊天应用”，也不是一个“自动写 Prompt 的代理层”，而是一个：

> **把显式技能、技能检索、技能草稿生成、技能治理与技能版本演化统一到本地文件工作台中的 Agent 能力操作系统。**
