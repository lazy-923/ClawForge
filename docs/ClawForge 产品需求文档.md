# ClawForge 产品需求文档（PRD）

## 一、项目名称与定位

### 1. 项目名称

**ClawForge**

### 2. 名称含义

`Claw` 延续 OpenClaw 一系本地 Agent 的显式技能脉络；`Forge` 强调“锻造、提炼、演化”，体现本项目不再停留于静态 Skill 文件，而是把交互经验逐步锻造成可复用、可治理、可演化的技能资产。

### 3. 项目一句话定义

ClawForge 是一个面向本地 AI Agent 的 **Skill Gateway + Skill Evolution** 工作台，用文件驱动的方式把“技能发现、技能激活、技能草稿生成、技能合并提升”串成一条透明、可编辑、可审查的闭环。

---

## 二、项目背景

### 1. 当前问题

ClawForge 当前已经具备一套本地 Agent 基础能力：

- 文件驱动 Prompt
- `SKILL.md` 技能机制
- `MEMORY.md` 长期记忆
- Knowledge 检索
- 工具调用可视化
- 本地会话持久化
- 前端工作台

这些能力已经足以支撑一个可运行的本地 Agent 工作流，但如果只停留在这一层，Skill 机制仍然存在明显短板：

1. **技能激活方式过于被动**
- 当前主要依赖主 Agent 阅读 `SKILLS_SNAPSHOT.md` 后自行决定要不要读取某个 `SKILL.md`
- 缺少独立的 Skill Gateway，无法对技能做专门的检索、筛选和注入

2. **技能库是静态的**
- 现在的技能主要靠手工编写
- 用户长期交互中暴露出的稳定偏好、约束和流程，无法沉淀为显式技能资产

3. **经验沉淀停留在 memory 层**
- 当前系统能记住信息，但难以把“以后应该怎么做”升级成结构化、可执行的行为规范

4. **缺少技能治理**
- 缺少技能草稿、版本、来源、合并、淘汰等机制
- Skill 数量一多，Prompt 会膨胀，技能也容易重复与污染

### 2. 引入新方向的目标

本项目不是另起一个独立系统，而是要把现有本地 Agent 基线能力与更完整的技能中枢能力合并为同一个 ClawForge：

- 让 Skill 从“静态说明书”升级为“动态可演化工件”
- 让技能激活从“主 Agent 自行猜测”升级为“Gateway 检索与注入”
- 让经验沉淀从“原始记忆片段”升级为“技能草稿与技能资产”
- 让聊天、会话、Prompt、Memory、Knowledge、Tools、Skill Gateway、Skill Evolution 和前端工作台成为一套统一产品，而不是拆成前后两段项目叙事

---

## 三、产品目标

### 1. 总目标

打造一个 **本地优先、文件驱动、透明可控** 的 Agent 技能工作台，使系统具备以下能力：

1. 根据当前请求自动检索最相关技能，而不是把全部技能硬塞进 Prompt
2. 从用户交互中自动生成候选技能草稿
3. 对技能草稿执行 add / merge / ignore 的治理流程
4. 保持整个技能生命周期可观察、可编辑、可回滚

### 2. 非目标

本阶段明确不做：

- 自动无人值守地全量改写主技能库
- 复杂多租户 SaaS 平台
- 依赖外部向量数据库的重型架构
- 把项目做成单纯的 OpenAI-compatible 代理服务

---

## 四、核心产品理念

### 1. 文件优先，而不是黑盒优先

所有关键状态优先存为本地文件，而不是隐含在系统内部：

- 正式技能
- 技能草稿
- 技能版本历史
- 技能使用统计
- 技能合并记录

### 2. 显式技能，而不是隐式 Prompt 漂移

系统不是偷偷修改系统提示词，而是把能力沉淀为显式工件，允许用户：

- 查看
- 编辑
- 对比
- Promote
- 回滚

### 3. 半自动演化，而不是全自动污染

ClawForge 不追求“每轮对话自动入库”，而是追求：

```text
交互信号 -> 生成 Skill Draft -> 提供治理建议 -> 人工确认 Promote
```

这保证系统既能学习，又不至于失控。

### 4. 本地工作台，而不是纯后端代理

ClawForge 的最终形态是一个本地 Agent 工作台：

- 左侧管理会话与技能
- 中间进行对话与观察技能命中
- 右侧审查和编辑 Skill Draft / Skill / Memory

---

## 五、目标用户

### 1. 第一类用户：Prompt / Agent 开发者

特点：

- 需要不断调试 Agent 行为
- 希望技能体系可审查、可维护
- 不满足于黑盒式“记忆增强”

需求：

- 想知道本轮为什么命中了某个技能
- 想知道某条新技能是从哪轮对话提炼出来的
- 想手工编辑技能，而不是完全信任自动生成

### 2. 第二类用户：重度本地 AI 工作流使用者

特点：

- 长期重复做类似任务
- 有稳定风格、约束和偏好
- 希望 Agent 越用越懂自己

需求：

- 系统能积累“做事方式”，不只是记住“历史内容”
- 不想每次重新描述流程与格式要求

## 六、核心能力范围

## 6.1 记忆系统

### 功能目标

为 Agent 提供一套本地优先、文件优先、可审查的上下文记忆体系，使系统既能保留当前对话上下文，也能跨会话保留长期记忆，并在需要时检索知识内容。

### 记忆系统应包含的三层结构

#### 1. Session Memory

即当前会话历史。

职责：

- 保存当前 session 的消息列表
- 为同一会话中的后续轮次提供直接上下文
- 支撑标题生成、会话切换、前端消息回放

设计要求：

- 每个 session 独立持久化
- 会话历史必须人类可读
- 会话消息应保留用户消息、助手消息和必要的工具调用轨迹

#### 2. Long-term Memory

即全局长期记忆。

职责：

- 保存跨会话仍然稳定成立的信息
- 保存用户偏好、长期约束、常驻身份信息
- 在每轮对话时作为额外上下文补充，而不是替代 session history

设计要求：

- 长期记忆优先存为 `MEMORY.md`
- 可直接编辑
- 可通过检索动态注入当前请求
- 检索结果不应无控制地写回会话历史，避免会话文件膨胀

#### 3. Knowledge Memory

即本地知识库。

职责：

- 提供对 PDF / Markdown / TXT 等知识文档的检索
- 解决“查资料”“查文档”“查知识内容”类问题
- 与长期记忆区分开：长期记忆偏用户与系统长期状态，知识库偏外部资料和参考内容

设计要求：

- 从本地 `knowledge/` 目录构建索引
- 通过工具按需检索，而不是默认全量注入
- 支持后续扩展到离线文档提炼技能

### 产品要求

ClawForge 的记忆系统不是单一的“记忆增强”，而是三种上下文的协作：

1. 当前 session 的直接历史
2. 全局长期记忆
3. 可检索知识库

最终目标是：

- 会话上下文负责“这次聊到哪了”
- 长期记忆负责“长期稳定成立什么”
- 知识检索负责“外部资料里有什么”

### 产品价值

- 降低用户重复描述背景与偏好的次数
- 让 Agent 对长期风格和约束更稳定
- 让知识检索和会话历史各司其职
- 保持所有关键上下文来源可见、可改、可追踪

## 6.2 核心工具

### 功能目标

为 Agent 提供一组稳定、通用、可审查的基础执行能力，使技能文件和 Prompt 不只是“描述应该怎么做”，而是能真正驱动系统去行动。

### 设计原则

ClawForge 的技能遵循“指令遵循”而不是“函数硬编码”的方式：

- Skill 本身不是 Python 函数
- Skill 是告诉 Agent 如何组合使用基础工具完成任务的说明书
- Agent 通过读取技能内容，再动态调用核心工具执行

这意味着核心工具是系统能力的最底层公共基座。

### 当前应内置的核心工具

#### 1. `terminal`

功能：

- 在项目工作区内执行终端命令

适用场景：

- 文件系统操作
- 命令行调试
- 调用本地 CLI

产品要求：

- 必须限制工作目录
- 必须有危险命令防护
- 必须限制超时和输出长度

#### 2. `python_repl`

功能：

- 执行短 Python 代码

适用场景：

- 文本转换
- 结构化处理
- 简短计算与数据分析

产品要求：

- 应与终端工具职责分离
- 更适合小规模脚本与临时推导

#### 3. `fetch_url`

功能：

- 抓取网页或 HTTP 资源，并转为适合模型处理的文本

适用场景：

- 查网页
- 获取 API / JSON 内容
- 读取在线资料

产品要求：

- 需要有超时与输出截断
- HTML 内容应转为易读文本，而不是把原始网页源码直接交给模型

#### 4. `read_file`

功能：

- 读取项目目录内文件

适用场景：

- 读取技能文件
- 读取工作区文件
- 读取 Memory / Knowledge / 配置文件

产品要求：

- 必须限制路径遍历
- 必须防止逃逸出项目根目录

#### 5. `search_knowledge_base`

功能：

- 对本地知识库执行检索

适用场景：

- 查找具体知识内容
- 从文档中提取依据
- 作为回答和技能抽取的外部资料来源

产品要求：

- 应支持本地索引持久化
- 应以检索方式按需返回片段，而不是直接读完整知识库

### 工具系统的产品价值

- 让技能成为“可执行工作流”，而不是静态说明
- 保持能力边界清晰：检索、读取、执行、抓取各自独立
- 让 Agent 的行动过程可以被观察和回放

## 6.3 Skill Gateway

### 功能目标

在每轮对话前，先通过一条专门的技能激活路径，决定应该给主 Agent 注入哪些技能。

### 核心能力

1. Query Rewrite
- 将用户问题改写为更适合检索技能的查询

2. Skill Retrieval
- 从正式技能库中检索候选技能

3. Skill Selection
- 判断候选技能是否真的适合当前任务

4. Skill Context Injection
- 将少量相关技能注入到 Prompt，而不是全量注入

### 产品价值

- 降低 Prompt 膨胀
- 提升技能命中精度
- 让 Skill 激活过程可解释

---

## 6.4 Skill Drafting

### 功能目标

从真实交互中提取可复用的行为规则，生成候选技能草稿。

### 抽取原则

只提取：

- 稳定要求
- 可复用流程
- 可迁移的风格或约束

不提取：

- 一次性请求
- 临时上下文
- 具体 case 的实体内容

### 输出形态

Skill Draft 应至少包含：

- 名称
- 描述
- Goal
- Constraints / Style
- 可选 Workflow
- 来源会话
- 置信度
- 推荐动作

---

## 6.5 Skill Governance

### 功能目标

对新技能草稿进行治理，而不是直接写入正式技能库。

### 治理动作

1. `add`
- 新增为一项正式技能

2. `merge`
- 合并到已有技能中

3. `ignore`
- 放弃该草稿
### 治理要求

- 给出推荐动作
- 给出相似技能
- 给出为什么要 add / merge / ignore
- 保留人工确认入口

---

## 6.6 Skill Versioning

### 功能目标

让正式技能具备版本历史，而不是每次被覆盖后丢失上下文。

### 核心能力

- 记录 skill lineage
- 记录 merge history
- 支持 patch 级版本递增
- 支持查看前后差异
- 支持回滚

---

## 6.7 Skill Workspace

### 功能目标

把技能系统做成一个可视化工作台，而不是只靠文件夹操作。

### 工作台视图

1. 正式技能列表
2. 技能草稿列表
3. 本轮命中技能
4. 技能版本历史
5. 技能使用统计

---

## 七、关键产品差异化

### 1. 与传统 Memory 系统的区别

Memory 记住的是：

- 事实
- 偏好
- 历史片段

ClawForge 关注的是：

- 行为规范
- 工作流策略
- 可执行技能

### 2. 与纯 Prompt 管理系统的区别

纯 Prompt 管理往往只能编辑静态提示词。

ClawForge 进一步解决：

- 技能检索
- 技能激活
- 技能草稿抽取
- 技能版本演化

### 3. 与 Auto-Skill 的区别

ClawForge 吸收的是“技能演化思想”，但产品形态不同：

1. 以本地工作台为中心，而不是以代理服务为中心
2. 以文件工件为第一存储单元，而不是以运行时服务为第一中心
3. 采用半自动 Promote 机制，而不是默认全自动闭环
4. 更强调前端可视化审查和手工编辑能力

## 八、功能需求

## 8.1 P0 功能

### P0-1 技能网关主链路

- 接收用户消息
- 先做技能查询改写
- 检索相关技能
- 将命中技能注入 Prompt
- 返回命中结果给前端

### P0-2 技能草稿生成

- 每轮对话结束后异步抽取 Skill Draft
- 写入本地 `skill_drafts/`

### P0-3 技能治理建议

- 对每个 Draft 给出：
  - 推荐动作
  - 相似技能
  - 理由说明

### P0-4 前端可观察性

- 显示本轮命中的技能
- 显示新生成的 Skill Draft
- 可以打开并查看 Draft 内容

---

## 8.2 P1 功能

### P1-1 Skill Promote

- 用户可手动将 Draft Promote 为正式技能

### P1-2 Skill Merge

- 用户可将 Draft 合并到相似技能

### P1-3 Skill Versioning

- 记录每次 Promote / Merge 的版本历史

### P1-4 使用统计

- 记录技能被检索次数
- 记录技能被采纳次数

---

## 8.3 P2 功能

### P2-1 技能淘汰建议

- 对长期高召回低使用的技能给出 stale 提示

### P2-2 文档到技能

- 从知识文档中离线提取候选技能

### P2-3 Replay 驱动技能评估

- 对高频技能做离线回放评估

---

## 九、关键用户流程

## 9.1 在线问答流程

```text
用户输入问题
-> Skill Gateway 改写查询
-> 检索技能
-> 选择并注入技能
-> 主 Agent 执行
-> 返回回答 + 命中技能
```

## 9.2 技能沉淀流程

```text
一轮对话结束
-> Draft Extractor 提取候选技能
-> Skill Judge 生成 add / merge / ignore 建议
-> 写入 skill_drafts/
-> 前端展示
-> 用户确认 Promote / Merge / Ignore
```

## 9.3 技能演化流程

```text
用户打开某个 Draft
-> 查看推荐相似技能
-> 执行 Promote 或 Merge
-> 更新正式技能库
-> 写入版本与历史记录
```

## 十、成功标准

### 1. P0 成功标准

1. 系统能在聊天前命中并注入少量相关技能
2. 前端能展示本轮命中技能
3. 对话结束后能生成 Skill Draft
4. Draft 可以被查看和审查

### 2. P1 成功标准

1. Draft 可以 Promote 为正式技能
2. Draft 可以 Merge 到已有技能
3. 正式技能保留版本历史

### 3. 产品层成功标准

1. 用户重复描述流程和风格约束的次数明显下降
2. 技能命中质量优于“全量技能快照 + Agent 自行猜测”
3. 技能库能持续增长，但不明显失控或污染

## 十一、阶段规划

### 阶段一：Skill Gateway MVP

- Query Rewrite
- Skill Retrieval

### 阶段二：Skill Draft MVP

- Draft Extraction

- add / merge / ignore 建议

- stale skill 审计

- 离线评估

## 十二、最终产品描述

ClawForge 不是一个“带记忆的聊天应用”，也不是一个“自动写 Prompt 的代理层”，而是一个：

> **把显式技能、技能检索、技能草稿生成、技能治理与技能版本演化统一到本地文件工作台中的 Agent 能力操作系统。**
